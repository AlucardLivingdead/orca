Orca Bug Dependency List

This page contains a list of bugs filed against external components and applications. This allows us to better track which releases of external components that Orca will work with and also the behavior one might experience in the presence of the bug.

NOTE: this list is not always up to date. Some of the bugs may have already been fixed by the component developers (THANKS!).

  • Evolution Bugs

  • Firefox/Thunderbird Bugs

  • OpenOffice Bugs

Acroread 7.0.8+

Accessible hierarchy is incomplete

No

No

SECTION 508 AND USABILITY PROBLEM - MUST BE FIXED
There are a number of objects, mostly drawing areas, in the acroread hierarchy for which we do not have complete information. Because these objects cannot be accessed via AT-SPI, we cannot reliably examine a hierarchy in which they are found, obtain role information, obtain a child count, etc. As a result, it is difficult to determine if focus is in a particular object and to obtain information about siblings within such an object (e.g. to provide access to text which is serving as a label for controls within the Search panel. See below.).
Many times this can be verified using AT-POKE: These objects are shown without any state information.

Acroread 7.0.8+

Acroread does not generate an event when text selection changes

No

No

SECTION 508 AND USABILITY PROBLEM - MUST BE FIXED
Unable to accurately notify the user about what text has been selected/unselected, making it difficult for users to accurately and efficiently select and copy a portion of a document.
This bug can be reproduced by logging "object:text-selection-changed" events in AT-POKE and then attempting to select text in acroread.

Acroread 7.0.8+

Acroread does not expose selected text through getNSelections

No

No

SECTION 508 AND USABILITY PROBLEM - MUST BE FIXED
Unable to accurately notify the user about what text is currently selected/unselected, making it difficult for users to accurately and efficiently select and copy a portion of a document and/or get reoriented to the screen contents and state.
This bug can be reproduced using the acroread_diag.py script.

Acroread 7.0.8+

Acroread does not respond to setCaretOffset

No

No

SECTION 508 AND USABILITY PROBLEM - MUST BE FIXED

  1. Unable to position the caret when a "say all" has been interrupted/completed, making it difficult to read a large document in chunks (e.g. by paragraph or page).

  2. Unable to position the caret when a braille user presses a cursor routing key on his/her display.

This bug can be reproduced using the acroread_diag.py script as well as by using AT-POKE.

Acroread 7.0.8+

Caret has no relation to one's location when tabbing

No

No

SECTION 508 AND USABILITY PROBLEM - MUST BE FIXED
When in a document which contains links to other sections of the document, pressing Tab should move the user to the next link with respect to the (selection) caret and not with respect to the top of the page as it currently does. Similarly, having tabbed to a new location, the caret should be moved to that same location rather than left in place -- as is currently the case. The present functionality is disorienting to the user who is accessing the content non-visually. It also slows the user down because it is necessary to manually "sync" one's locations.
This bug can be reproduced without the need for special tools.

Acroread 7.0.8+

Acroread does not expose any text attribute information

No

No

SECTION 508 AND USABILITY PROBLEM - MUST BE FIXED
Unable to provide any information about the appearance of the document, making it necessary for a user who needs this information to seek sighted assistance.
This bug can be reproduced using the acroread_diag.py script.

Acroread 7.0.8+

Acroread does not expose any size and position information for the character/glyph at a given offset

No

No

SECTION 508 AND USABILITY PROBLEM - MUST BE FIXED
Users cannot use flat review to examine a document. In addition, we have no way to check for the case where text is contained in multiple drawing areas but spatially occupying a single line. When that occurs, we can only provide access to the text in the drawing area which contains the caret.
This bug can be reproduced using the acroread_diag.py script.

Acroread 7.0.8+

Acroread does not generate events when navigating within the text of an image.

No

No

SECTION 508 AND USABILITY PROBLEM - MUST BE FIXED
Some documents present information in an image whose name contains the full text being displayed. Please see this document from census.gov. In these circumstances one can use the keyboard to navigate within, as well as select, the content. However, without the corresponding events, we cannot provide access to the text.
This bug can be reproduced by logging events in AT-POKE and then using the keyboard to navigate and select text within the document.

Acroread 7.0.8+

Pages panel is inaccessible: fails to provide relevant events; generates erroneous object:text-caret-moved events

No

No

USABILITY PROBLEM - SHOULD BE FIXED
When navigating among the thumbnails in the Pages panel we fail to receive any relevant events (e.g. object:selection-changed). Instead, each press of an arrow key results in an object:text-caret-moved for the text which had focus prior to our moving from the document to the Pages panel. As a result, the user cannot access the Pages panel because we cannot identify what page he/she has arrowed to.
Work-around: Use the "Go to page" dialog instead.
This bug can be reproduced by logging events in AT-POKE and then using the keyboard to navigate to, and then within, the Pages panel.

Acroread 7.0.8+

Acroread should provide additional options for keyboard navigation within tables.

No

No

USABILITY PROBLEM - SHOULD BE FIXED
In a typical table within Acrobat Reader, using the arrow keys moves you from cell to cell horizontally first. This makes it quite tedious and time-consuming to navigate within a table if you cannot use a mouse. There should be commands to:

  • Move to the next/previous row

  • Move to the beginning/end of the row

  • Move to the beginning/end of the table

Note: This cannot be remedied within Orca due to the fact that acroread does not respond to setCaretOffset.
This bug can be reproduced using the acroread_diag.py script.

Acroread 7.0.8+

Text boxes in the Search Panel should be labeled

No

No

USABILITY PROBLEM - SHOULD BE FIXED
The text boxes are not labeled, but preceded in the drawing area by an object of role "unknown" which contains the text that should be the label.
Note: We cannot work around this situation because the parent which contains the text box and "label" is of type None (thus we cannot access its children).
This bug can be reproduced using AT-POKE.

Acroread 7.0.8+

It is not possible to access the page number text box in the Acroread tool bar

No

No

USABILITY PROBLEM - SHOULD BE FIXED
The text box near the bottom of the Acrobate reader window cannot be given focus without using the mouse. In addition, its contents cannot be accessed via AT-SPI. As a result, there is no way to notify the user of his/her new location when navigating by page or after performing a search.
Work around: The user can manually get into the GoTo Page dialog to obtain the current page and then use flat review to obtain the total number of pages, but fixing this problem would make the user much more efficient.
This bug can be reproduced using AT-POKE.

Acroread 7.0.8+

Acroread does not implement RelationType

No

No

NICE TO FIX
Providing the relationship between objects (e.g. RELATION_FLOWS_TO, RELATION_FLOWS_FROM, RELATION_LABEL_FOR, RELATION_LABELLED_BY, etc.) would take the guesswork out of providing access to the information in the acroread window because we would know how one object related to another rather than having to surmise relationships based on observation.
This bug can be reproduced using AT-POKE.
